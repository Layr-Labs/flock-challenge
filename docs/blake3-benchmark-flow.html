Flock · BLAKE3 · Apple Silicon

A benchmark you can review in one sitting.

One editable prover makes one proof. One committed verifier binary chooses the private blocks, owns the clock, checks the proof, and writes the score.

The entire security argument

The candidate never chooses the scored input and never participates in verification or scoring.

01 / FLOW

Six steps, no hidden service.

Each ranked trial starts a fresh prover process. The protected verifier binary remains in control from private-input creation through scoring.

01

Prepare

The trusted verifier binary starts the prover inside the macOS sandbox.

02

Warm

The prover builds setup and runs one seed-independent warm-up.

03

Challenge

After readiness, the verifier draws a fresh 64-bit seed and starts its clock.

04

Prove

The prover expands the private blocks, calls prove_fast, writes the proof, and exits.

05

Verify

After timing stops, the binary rebuilds the same witness, checks its commitment, then verifies.

06

Score

Only after every trial passes, the verifier writes score.json.

02 / TRUST

Broad optimization. Fixed judgment.

The editable surface is deliberately large. Isolation comes from committing a checksum-protected judge built against a permanently pinned source tree—not from pretending shared verifier imports are safe.

Why not verify against the candidate's parent? A winning submission may later become the baseline, and the editable trees contain normal verifier source. Parent-derived trust would therefore drift after promotion. The fixed reviewed source commit stays sharp across every future submission.
03 / CLOCK

The seed crosses the boundary after the clock starts.

Setup and warm-up are excluded, matching the intent of the upstream benchmark. Work depending on the scored input, including serialization, is included.

UNTIMED
Create fresh prover process → initialize Flock → create Blake3Setup → warm up with a fixed unrelated seed → write ready file.
START CLOCK
Verifier draws a private seed from /dev/urandom, records Instant::now(), then writes it to prover stdin. The seed compactly defines all 65,536 test blocks.
TIMED
Generate 65,536 inputs → generate witness → commit → prove → serialize → write proof → prover exits.
STOP CLOCK
The verifier observes prover exit. A forged early ready file only starts this clock too early and cannot improve the score.
MUST PASS
Bounded decode → trusted input and witness reconstruction → commitment equality → pristine verifier.
WRITE SCORE
score = 65,536 / min(trial_seconds), after all three trials verify.

05 / VERIFY

A proof passes six independent gates.

The protocol verifier alone does not bind this benchmark's private input. The trusted commitment reconstruction supplies that missing statement binding.

Prover exited

The timer does not stop on a message or partial file. It stops only after successful process exit.

Size bounded

The proof must be nonempty and no larger than 50 MiB before allocation and decoding.

Format exact

Magic, version, R1CS flavor, fixed-width encoding, and end-of-input all must match.

Input rebuilt

The trusted generator expands the undisclosed seed into the exact BLAKE3 tuples.

Commitment equal

Root and every PCS parameter must equal a fresh commitment to the pristine witness.

Proof valid

The immutable original Blake3Setup::verify runs under the fixed transcript domain.

08 / REVIEW

Suggested 20-minute review order.

Start with policy, then trace the tiny candidate program, then spend the remaining time on the trusted trial and verification functions.

  1. benchmark.jsonConfirm the only editable paths, score path, direction, commands, and workflow.
  2. common/src/lib.rsConfirm deterministic input generation and fixed transcript domain.
  3. worker/src/main.rsRead the complete untrusted wrapper: setup, warm-up, seed, prove, serialize, exit.
  4. harness/src/main.rsFocus on run_trial, verify_proof, and deserialize_bundle.
  5. trusted/SHA256SUMSConfirm the committed arm64 verifier has the reviewed protected hash.
  6. build-trusted-verifier.shConfirm the author-only build pins the reviewed source commit, toolchain, and conservative CPU target.
  7. setup.shConfirm prerequisite checks, TLS Rustup bootstrap, pinned toolchain, fetch retry, binary integrity, and offline candidate build.
  8. benchmark.shConfirm score deletion, scratch-only writes, Seatbelt rules, and harness arguments.
  9. workflowConfirm exact SHA, Setup/Benchmark steps, fail-closed upload, and pinned Actions.
